# 基于用户反馈的技能改进

**日期：** 2025-11-28
**状态：** 草案
**来源：** 两个在实际开发场景中使用 superpowers 的 Claude 实例

---

## 执行摘要

两个 Claude 实例从实际开发会话中提供了详细反馈。他们的反馈揭示了当前技能中的**系统性缺陷**，这些缺陷导致可预防的 bug 在遵循技能指导的情况下仍然被发布。

**关键洞察：** 这些是问题报告，而不仅仅是解决方案提案。问题是真实的；解决方案需要仔细评估。

**关键主题：**
1. **验证缺口** - 我们验证操作成功，但不验证它们是否达到预期结果
2. **进程卫生** - 后台进程积累并在子代理之间相互干扰
3. **上下文优化** - 子代理获得太多无关信息
4. **缺少自我反思** - 在交接前没有提示批评自己的工作
5. **Mock 安全性** - Mock 可能与接口不同步而未被检测到
6. **技能激活** - 技能存在但未被阅读/使用

---

## 已识别的问题

### 问题 1：配置变更验证缺口

**发生了什么：**
- 子代理测试"OpenAI 集成"
- 设置了 `OPENAI_API_KEY` 环境变量
- 获得状态 200 响应
- 报告"OpenAI 集成正常工作"
- **但是** 响应包含 `"model": "claude-sonnet-4-20250514"` - 实际上使用的是 Anthropic

**根本原因：**
`verification-before-completion` 检查操作成功，但不检查结果是否反映预期的配置变更。

**影响：** 高 - 对集成测试的错误信心，bug 发布到生产环境

**示例失败模式：**
- 切换 LLM 提供商 → 验证状态 200 但不检查模型名称
- 启用功能标志 → 验证无错误但不检查功能是否激活
- 更改环境 → 验证部署成功但不检查环境变量

---

### 问题 2：后台进程积累

**发生了什么：**
- 会话期间派发了多个子代理
- 每个都启动了后台服务器进程
- 进程积累（4+ 个服务器在运行）
- 过时进程仍然绑定端口
- 后来的 E2E 测试命中了配置错误的过时服务器
- 混乱/不正确的测试结果

**根本原因：**
子代理是无状态的 - 不知道之前子代理的进程。没有清理协议。

**影响：** 中高 - 测试命中错误服务器，假通过/失败，调试困惑

---

### 问题 3：子代理提示中的上下文膨胀

**发生了什么：**
- 标准方法：给子代理完整的计划文件阅读
- 实验：只给任务 + 模式 + 文件 + 验证命令
- 结果：更快，更专注，单次尝试完成更常见

**根本原因：**
子代理在无关计划部分浪费 token 和注意力。

**影响：** 中等 - 执行更慢，失败尝试更多

**有效的做法：**
```
You are adding a single E2E test to packnplay's test suite.

**Your task:** Add `TestE2E_FeaturePrivilegedMode` to `pkg/runner/e2e_test.go`

**What to test:** A local devcontainer feature that requests `"privileged": true`
in its metadata should result in the container running with `--privileged` flag.

**Follow the exact pattern of TestE2E_FeatureOptionValidation** (at the end of the file)

**After writing, run:** `go test -v ./pkg/runner -run TestE2E_FeaturePrivilegedMode -timeout 5m`
```

### 问题 4：交接前没有自我反思

**发生了什么：**
- 添加了自我反思提示："用新鲜的眼光审视你的工作 - 哪些可以做得更好？"
- 任务 5 的实现者识别出测试失败是由于实现 bug，而不是测试 bug
- 追溯到第 99 行：`strings.Join(metadata.Entrypoint, " ")` 创建了无效的 Docker 语法
- 没有自我反思，只会报告"测试失败"而不提供根本原因

**根本原因：**
实现者在报告完成之前不会自然地退后一步批评自己的工作。

**影响：** 中等 - Bug 被交给审查者，而实现者本可以自己发现

### 问题 5：Mock 与接口漂移

**发生了什么：**
```typescript
// 接口定义 close()
interface PlatformAdapter {
  close(): Promise<void>;
}

// 代码（有 BUG）调用 cleanup()
await adapter.cleanup();

// Mock（匹配 BUG）定义 cleanup()
vi.mock('web-adapter', () => ({
  WebAdapter: vi.fn().mockImplementation(() => ({
    cleanup: vi.fn().mockResolvedValue(undefined),  // 错误！
  })),
}));
```
- 测试通过
- 运行时崩溃："adapter.cleanup is not a function"

**根本原因：**
Mock 来源于有 bug 的代码调用的内容，而不是接口定义。TypeScript 无法捕获具有错误方法名的内联 mock。

**影响：** 高 - 测试给出错误信心，运行时崩溃

**为什么 testing-anti-patterns 没有阻止这个：**
该技能涵盖了测试 mock 行为和不理解就 mock，但没有涵盖"从接口而非实现派生 mock"这一特定模式。

### 问题 6：代码审查者文件访问

**发生了什么：**
- 派发了代码审查者子代理
- 无法找到测试文件："该文件似乎不存在于仓库中"
- 文件实际存在
- 审查者不知道要先明确读取它

**根本原因：**
审查者提示不包含明确的文件读取说明。

**影响：** 低中 - 审查失败或不完整

---

### 问题 7：修复工作流延迟

**发生了什么：**
- 实现者在自我反思期间识别出 bug
- 实现者知道修复方法
- 当前工作流：报告 → 我派发修复者 → 修复者修复 → 我验证
- 额外往返增加延迟但不增加价值

**根本原因：**
当实现者已经诊断出问题时，实现者和修复者角色之间的刚性分离。

**影响：** 低 - 延迟，但无正确性问题

---

### 问题 8：技能未被阅读

**发生了什么：**
- `testing-anti-patterns` 技能存在
- 人类和子代理在编写测试前都没有阅读它
- 本可以防止一些问题（但不是全部 - 见问题 5）

**根本原因：**
没有强制子代理阅读相关技能。没有提示包含技能阅读。

**影响：** 中等 - 如果不使用，技能投资就浪费了

## 建议的改进

### 1. verification-before-completion：添加配置变更验证

**添加新部分：**

```markdown
## 验证配置变更

当测试配置、提供商、功能标志或环境的变更时：

**不要只验证操作成功。验证输出反映预期的变更。**

### 常见失败模式

操作成功是因为*某些*有效配置存在，但它不是您打算测试的配置。

### 示例

| 变更 | 不充分 | 必需 |
|--------|-------------|----------|
| 切换 LLM 提供商 | 状态 200 | 响应包含预期的模型名称 |
| 启用功能标志 | 无错误 | 功能行为实际激活 |
| 更改环境 | 部署成功 | 日志/变量引用新环境 |
| 设置凭证 | 认证成功 | 认证的用户/上下文正确 |

### 门控函数

```
在声称配置变更有效之前：

1. 识别：此变更后应该有什么不同？
2. 定位：在哪里可以观察到这个差异？
   - 响应字段（模型名称、用户 ID）
   - 日志行（环境、提供商）
   - 行为（功能激活/未激活）
3. 运行：显示可观察差异的命令
4. 验证：输出包含预期的差异
5. 然后才：声称配置变更有效

红旗：
  - "请求成功"但未检查内容
  - 检查状态码但不检查响应体
  - 验证无错误但没有正面确认
```

**为什么有效：**
强制验证意图，而不仅仅是操作成功。

### 2. subagent-driven-development：为 E2E 测试添加进程卫生

**添加新部分：**

```markdown
## E2E 测试的进程卫生

当派发启动服务（服务器、数据库、消息队列）的子代理时：

### 问题

子代理是无状态的 - 它们不知道之前子代理启动的进程。后台进程持续存在并可能干扰后续测试。

### 解决方案

**在派发 E2E 测试子代理之前，在提示中包含清理：**

```
在启动任何服务之前：
1. 终止现有进程：pkill -f "<service-pattern>" 2>/dev/null || true
2. 等待清理：sleep 1
3. 验证端口空闲：lsof -i :<port> && echo "ERROR: Port still in use" || echo "Port free"

测试完成后：
1. 终止您启动的进程
2. 验证清理：pgrep -f "<service-pattern>" || echo "Cleanup successful"
```

### 示例

```
任务：运行 API 服务器的 E2E 测试

提示包含：
"在启动服务器之前：
- 终止任何现有服务器：pkill -f 'node.*server.js' 2>/dev/null || true
- 验证端口 3001 空闲：lsof -i :3001 && exit 1 || echo 'Port available'

测试后：
- 终止您启动的服务器
- 验证：pgrep -f 'node.*server.js' || echo 'Cleanup verified'"
```

### 为什么重要

- 过时进程使用错误配置响应请求
- 端口冲突导致静默失败
- 进程积累拖慢系统
- 混乱的测试结果（命中错误服务器）
```

**权衡分析：**
- 增加提示样板
- 但防止非常混乱的调试
- 对于 E2E 测试子代理值得

### 3. subagent-driven-development：添加精简上下文选项

**修改步骤 2：使用子代理执行任务**

**之前：**
```
Read that task carefully from [plan-file].
```

**之后：**
```
## 上下文方法

**完整计划（默认）：**
当任务复杂或有依赖时使用：
```
从 [plan-file] 仔细阅读任务 N。
```

**精简上下文（用于独立任务）：**
当任务独立且基于模式时使用：
```
您正在实现：[1-2 句任务描述]

要修改的文件：[确切路径]
要遵循的模式：[对现有函数/测试的引用]
要实现的内容：[具体需求]
验证：[要运行的确切命令]

[不要包含完整计划文件]
```

**使用精简上下文当：**
- 任务遵循现有模式（添加类似测试、实现类似功能）
- 任务是自包含的（不需要其他任务的上下文）
- 模式引用足够（例如，"遵循 TestE2E_FeatureOptionValidation"）

**使用完整计划当：**
- 任务对其他任务有依赖
- 需要理解整体架构
- 需要上下文的复杂逻辑
```

**示例：**
```
精简上下文提示：

"您正在为 devcontainer features 的特权模式添加测试。

文件：pkg/runner/e2e_test.go
模式：遵循 TestE2E_FeatureOptionValidation（在文件末尾）
测试：元数据中带有 `"privileged": true` 的 Feature 导致 `--privileged` 标志
验证：go test -v ./pkg/runner -run TestE2E_FeaturePrivilegedMode -timeout 5m

报告：实现、测试结果、任何问题。"
```

**为什么有效：**
在适当时减少 token 使用，增加专注度，更快完成。

**修改步骤 2：使用子代理执行任务**

**添加到提示模板：**

```
完成后，在报告之前：

退后一步，用新鲜的眼光审视您的工作。

问自己：

**权衡：**
每个任务增加约 30 秒，但在审查前捕获问题。

### 6. testing-anti-patterns：添加 Mock-接口漂移反模式

**添加新的反模式 6：**

```markdown
## 反模式 6：从实现派生的 Mock

**违规：**
```typescript
// 代码（有 BUG）调用 cleanup()
await adapter.cleanup();

// Mock（匹配 BUG）有 cleanup()
const mock = {
  cleanup: vi.fn().mockResolvedValue(undefined)
};

// 接口（正确）定义 close()
interface PlatformAdapter {
  close(): Promise<void>;
}
```

**为什么这是错误的：**
- Mock 将 bug 编码到测试中
- TypeScript 无法捕获具有错误方法名的内联 mock
- 测试通过是因为代码和 mock 都是错误的
- 使用真实对象时运行时崩溃

**修复：**
```typescript
// ✅ 正确：从接口派生 mock

// 步骤 1：打开接口定义（PlatformAdapter）
// 步骤 2：列出那里定义的方法（close、initialize 等）
// 步骤 3：Mock 精确的那些方法

const mock = {
  initialize: vi.fn().mockResolvedValue(undefined),
  close: vi.fn().mockResolvedValue(undefined),  // 来自接口！
};

// 现在测试失败因为代码调用了不存在的 cleanup()
// 该失败在运行时之前揭示了 bug
```

### 门控函数

```
在编写任何 mock 之前：

  1. 停止 - 先不要看被测代码
  2. 查找：依赖项的接口/类型定义
  3. 阅读：接口文件
  4. 列出：接口中定义的方法
  5. Mock：只有那些方法，使用精确的名称
  6. 不要：看你的代码调用什么

  如果你的测试失败因为代码调用了 mock 中没有的东西：
    ✅ 好 - 测试发现了你代码中的 bug
    修复代码以调用正确的接口方法
    而不是 mock

  红旗：
    - "我会 mock 代码调用的东西"
    - 从实现复制方法名
    - 没有阅读接口就编写 Mock
    - "测试失败了所以我会把这个方法加到 mock 中"
```

**检测：**

当你看到运行时错误 "X is not a function" 而测试通过时：
1. 检查 X 是否被 mock 了
2. 比较 mock 方法和接口方法
3. 查找方法名不匹配
```

**为什么有效：**
直接解决反馈中的失败模式。

### 8. subagent-driven-development：允许实现者修复自己识别的问题

**修改步骤 2：**

**当前：**
```
子代理带着工作总结报告回来。
```

**建议：**
```
子代理执行自我反思，然后：

如果自我反思识别出可修复的问题：
  1. 修复问题
  2. 重新运行验证
  3. 报告："初始实现 + 自我反思修复"

否则：
  报告："实现完成"

报告中包含：
- 自我反思发现
- 是否应用了修复
- 最终验证结果
```

**为什么有效：**
当实现者已经知道修复方法时减少延迟。已记录案例：本可以为 entrypoint bug 节省一次往返。

**权衡：**
提示稍微复杂，但端到端更快。

## 开放问题

1. **精简上下文方法：**
   - 我们应该让它成为基于模式任务的默认吗？
   - 我们如何决定使用哪种方法？
   - 太精简遗漏重要上下文的风险？

2. **自我反思：**
   - 这会显著拖慢简单任务吗？
   - 应该只适用于复杂任务吗？
   - 我们如何防止"反思疲劳"变成例行公事？

3. **进程卫生：**
   - 这应该在 subagent-driven-development 中还是单独的技能？
   - 它是否适用于 E2E 测试之外的其他工作流？
   - 我们如何处理进程应该持续的情况（开发服务器）？

4. **技能阅读强制：**
   - 我们应该要求所有子代理阅读相关技能吗？
   - 我们如何防止提示变得太长？
   - 过度文档化和失去焦点的风险？

---

## 成功指标

我们如何知道这些改进有效？

1. **配置验证：**
   - 零次"测试通过但使用了错误配置"的实例
   - Jesse 不再说"这实际上不是在测试你认为的东西"

2. **进程卫生：**
   - 零次"测试命中错误服务器"的实例
   - E2E 测试运行期间没有端口冲突错误

3. **Mock-接口漂移：**
   - 零次"测试通过但运行时因缺少方法崩溃"的实例
   - mock 和接口之间没有方法名不匹配

4. **自我反思：**
   - 可衡量：实现者报告是否包含自我反思发现？
   - 定性：更少 bug 进入代码审查？

5. **技能阅读：**
   - 子代理报告引用技能门控函数
   - 代码审查中更少反模式违规

## 风险和缓解措施

### 风险：提示膨胀
**问题：** 添加所有这些要求使提示变得难以承受
**缓解：**
- 分阶段实现（不要一次添加所有内容）
- 使某些添加有条件（E2E 卫生只用于 E2E 测试）
- 考虑不同任务类型的模板

### 风险：分析瘫痪
**问题：** 太多反思/验证拖慢执行
**缓解：**
- 保持门控函数快速（秒，不是分钟）
- 精简上下文最初是选择加入
- 监控任务完成时间

### 风险：虚假的安全感
**问题：** 遵循清单不能保证正确性
**缓解：**
- 强调门控函数是最低限度，不是最高限度
- 在技能中保留"使用判断"语言
- 记录技能捕获常见失败，不是所有失败

### 风险：技能分歧
**问题：** 不同技能给出冲突的建议
**缓解：**
- 审查所有技能的变更以保持一致性
- 记录技能如何交互（集成部分）
- 部署前用真实场景测试

**立即进行第一阶段：**
- verification-before-completion：配置变更验证
- testing-anti-patterns：Mock-接口漂移
- requesting-code-review：明确的文件读取

**在最终确定前与 Jesse 测试第二阶段：**
- 获取自我反思影响的反馈
- 验证进程卫生方法
- 确认技能阅读要求值得开销

**等待验证后再进行第三阶段：**
- 精简上下文需要真实世界测试
- 实现者修复工作流变更需要仔细评估

这些变更解决了用户记录的真实问题，同时最小化使技能变差的风险。
