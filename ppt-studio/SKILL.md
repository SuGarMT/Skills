# PPT Studio — 一站式 HTML 演示文稿工作台

## 核心原则

1. **零依赖** — 输出单 HTML 文件，内联 CSS/JS，无需 npm/构建工具，双击即可演示
2. **阶段制推进** — 7 个阶段依次执行，每阶段有明确产出物和确认点
3. **进度追踪** — 通过 `_progress.md` 文件防止长对话遗忘（详见下方）
4. **展示而非口述** — 视觉主题通过浏览器预览选择，而非文字描述

## 进度文件机制（防遗忘）

**每轮对话开始时，先读取 `_progress.md` 确认当前阶段和下一步操作。每个阶段结束时，必须更新 `_progress.md`。**

进度文件保存在用户工作目录：`YYYYMMDD_主题/_progress.md`

```markdown
# PPT Studio 进度追踪

## 当前状态
- **阶段**: 2 (大纲讨论与修改)
- **大纲风格**: 乔布斯风 ✅
- **大纲文件**: ./大纲.md (12页)
- **已讨论页**: 1-5 / 共12页
- **视觉主题**: 待选
- **口播稿**: 需要

## 下一步
继续讨论第 6 页 → 全部确认后进入阶段 3: 打开 themes/index.html 选视觉主题

## 已完成
- [x] 阶段 0: 模式探测
- [x] 阶段 1: 需求确认 + 大纲生成
- [ ] 阶段 2: 大纲讨论与修改
- [ ] 阶段 3: 视觉主题选择
- [ ] 阶段 4: PPT 生成
- [ ] 阶段 5: 定制化视觉优化
- [ ] 阶段 6: 口播稿生成
- [ ] 阶段 7: 联合审核与优化
```

## 阶段 0: 模式探测

自动判断用户输入，决定起始阶段：

| 用户输入               | 起始阶段                      |
| ---------------------- | ----------------------------- |
| 一句话 / 主题          | → 阶段 1                     |
| 笔记 / 教程 / 素材文件 | → 阶段 1（素材作为内容基底） |
| 已有大纲 `.md`       | → 阶段 2（直接讨论修改）     |
| 已有 PPT `.html`     | → 阶段 5（定制化视觉优化）   |

**⚠ 阶段结束时：创建 `_progress.md`，记录检测结果和起始阶段。**

## 阶段 1: 需求确认 + 大纲生成

### 1.1 快速信息收集

一次性询问几个基础问题：

1. **演示目的**：教程讲解 / 技术分享 / 产品路演 / 内部汇报 / 知识科普 / 其他
2. **目标受众**：初学者 / 有基础 / 专业人士
3. **预期长度**：短 (5-8页) / 中 (10-15页) / 长 (16-20页)
4. **是否需要口播稿**：是 / 否

### 1.2 推荐大纲风格

**必须列出全部 5 种风格**，让用户看到完整选项。根据用户的目的、受众和素材特点，在推荐项后标注 `👈 推荐` 并附一句理由。用户可以选择任意风格，不限于推荐项。

**5 种大纲风格**：

| 风格                 | 文件                           | 核心特征                                                             |
| -------------------- | ------------------------------ | -------------------------------------------------------------------- |
| 🎯**乔布斯风** | `styles/jobs-style.md`       | 金句式标题（断言/比喻/对比，12字内），步骤化拆解，一句话打动人       |
| 📋**简洁风**   | `styles/minimal-style.md`    | 标题直述主题（如「环境配置」），精炼要点列表，零修辞，信息密度最高   |
| 📖**叙事风**   | `styles/narrative-style.md`  | 设问/场景引入（如「当代码第一次跑通…」），故事线展开，有情境感      |
| ⚡**行动风**   | `styles/action-style.md`     | 祈使句标题（如「三步搞定」），重操作步骤，少理论，面向动手型受众     |
| 🔬**分析风**   | `styles/analytical-style.md` | 结论式论断（如「成本下降40%的关键」），数据/证据驱动，善用对比和框架 |

确认后，读取对应的风格文件（`references/styles/xxx.md`）获取详细写作规则。

### 1.3 直接生成完整大纲

结合素材和已确认的风格规则，**一次性输出完整大纲**。

**产出物**：`YYYYMMDD_主题/大纲.md`

大纲格式（紧凑写法，每页标注推荐组件）：

```markdown
# 📊 PPT 大纲
> 风格: 乔布斯风 | 页数: 12 | 预估时长: ~8分钟

# 封面 [cover]
- series: AI 工具箱
- name: Skill 入门指南
- tagline: 触发、指令、工具链

# AI不是代替你，而是放大你 [title]
- subtitle: 核心概念篇

# 三步搞定 Skill 配置 [flow]
- 定义触发条件
- 编写指令文件
- 绑定工具链

# [动作指引: 实操演示穿插]
- 讲师此处切出 PPT，演示从零创建 Skill

# 感谢观看 [end]
```

**⚠ 阶段结束时：更新 `_progress.md`，记录大纲风格、页数、文件路径。下一步 → 阶段 2。**

## 阶段 2: 大纲讨论与修改

在已有大纲基础上逐页讨论修改。这是整个流程中最重要的协作环节——大纲质量直接决定最终 PPT 的质量。

### 讨论流程

#### 第一轮：整体结构审视

先展示完整大纲，从宏观角度讨论：

- 「整体分为 N 个章节，分别围绕 [A]、[B]、[C]。这个结构合理吗？需要增减章节吗？」
- 「页数是否合适？有没有感觉哪里太密集或太松散？」
- 如果用户对整体结构有调整，先在宏观层面改好，再进入逐页讨论。

#### 第二轮：逐页深入讨论

确认整体结构后，一页一页地过。AI 自行判断哪些页需要深入讨论（如内容页、关键转折页），简单页（封面、结束页）可以简短确认后跳过。

**每一页的讨论包含**：

1. 展示这一页的当前内容（标题 + 要点 + 组件类型）
2. 给出你的专业评估和修改建议（如果有的话）
3. 等待用户确认或提出修改意见
4. 确认后锁定，进入下一页

### 讨论原则

这些原则的目标是降低用户的认知负担、提高讨论效率，让不擅长设计的人也能做出好的决策。

#### 一次只讨论一页

不要同时抛出 3-5 页让用户批阅。一页讨论完、确认后，再进入下一页。这避免了信息过载，也让每次修改都能立刻落地。

#### 优先提供选择题而非开放式问题

「A 方案 vs B 方案，您倾向哪个？」比「这页您想怎么改？」更容易回答。

- ✅ "这页的标题，'三步搞定配置' 和 '从零开始配置环境'，您更喜欢哪个？"
- ✅ "这页用流程图(flow)还是要点列表(list)展示更合适？流程图强调步骤的先后顺序，列表更适合并列的要点。"
- ❌ "这页您觉得怎么样？"（太开放，用户不知道从哪个角度回答）

#### 主动推荐并解释理由

不要空问「想放什么」，而是给出建议后问「这样行不行」。你是专家，用户期望你引导方向。

- ✅ "这页我建议用**左右对比 (comparison)**，因为新旧方案的差异一目了然。您觉得合适吗？"
- ❌ "这页您想用什么组件？"

#### 当用户犹豫时，提供 2-3 个具体方案

如果某页的处理方式不明显，主动提出 2-3 个不同思路，每个思路附带一句话的优劣分析，然后给出你的推荐。

- "方案 A: 拆成两页——一页讲概念，一页讲操作（更清晰但页数+1）"
- "方案 B: 合在一页用左右分栏——左边概念右边操作（紧凑但信息密度高）"
- "我推荐方案 A，因为受众是初学者，拆开更容易消化。"

#### 灵活回退

用户说「前面第 3 页的标题我想改一下」，立刻回退处理，不要说「我们先往后走再回来改」。

#### 组件推荐有据可依

根据本文档中的**17 种组件类型参考表**进行推荐。每种组件有明确的适用场景，推荐时要解释为什么这种组件适合当前内容。

### 讨论示例

> **第 3 页 (内容页)**
>
> 当前内容：
>
> ```
> # 三步搞定 Skill 配置 [flow]
> - 定义触发条件
> - 编写指令文件
> - 绑定工具链
> ```
>
> 我觉得这页用**流程图 (flow)** 是对的——三个步骤有明确的先后顺序，flow 组件能清晰地展示这种线性关系。
>
> 不过，标题「三步搞定 Skill 配置」是行动风的写法。您确认的风格是乔布斯风，建议改为更有冲击力的表达，比如：
>
> - A: 「触发、指令、工具链——仅此三件」
> - B: 「一个文件，三个步骤，无限可能」
>
> 您觉得 A、B 哪个更好？还是保持原标题？

**⚠ 阶段结束时：更新 `_progress.md`，标记阶段 2 完成，记录讨论中的关键修改。下一步 → 阶段 3。**

## 17 种组件类型参考

讨论大纲和生成 PPT 时，必须参考此表进行组件选择和推荐。

| 组件     | class          | 适用内容                | 布局特征                      |
| -------- | -------------- | ----------------------- | ----------------------------- |
| 封面     | `cover`      | 第一页（固定）          | 主标题 + 系列名 + 标签 + 图标 |
| 章节标题 | `title`      | 章节过渡、核心观点强调  | 大标题 + 副标题，纯文字居中   |
| 核心观点 | `message`    | 一句话金句、核心论断    | 超大字号居中，极简背景        |
| 提问页   | `question`   | 引发思考、制造悬念      | 大问号 + 问题文字             |
| 左右分栏 | `split`      | 观点+数据并列、图文搭配 | 左侧文字 + 右侧卡片/数据      |
| 数据统计 | `stats`      | 3组并列核心数据展示     | 三列等宽数字卡片              |
| 网格卡片 | `card-grid`  | 4个并列特性/优势        | 2×2 网格布局                 |
| 代码展示 | `terminal`   | 代码片段、命令行操作    | 终端风格代码框                |
| 左右对比 | `comparison` | 新旧方案、正反对比      | 左右两栏对称结构              |
| 流程图   | `flow`       | 步骤 1-2-3 线性流转     | 垂直/水平步骤节点             |
| 时间线   | `timeline`   | 项目里程碑、历史沿革    | 时间轴 + 节点描述             |
| 引用     | `quote`      | 名人名言、用户反馈      | 大引号 + 引用文字 + 来源      |
| 图文说明 | `image-text` | 截图解说、产品展示      | 图片 + 文字并排               |
| 要点列表 | `list`       | 简单的文字要点平铺      | 编号/符号列表                 |
| 团队介绍 | `team`       | 人物卡片、团队展示      | 头像 + 姓名 + 角色            |
| 图片画廊 | `gallery`    | 多图并列展示            | 网格/并排图片                 |
| 结束页   | `end`        | 最后一页（固定）        | 致谢 + 联系方式               |

各组件的详细 HTML 结构和模板代码，见 [slide-types.md](references/slide-types.md)（阶段 4 生成 PPT 时读取）。

## 阶段 4: PPT 生成

**产出物**：`YYYYMMDD_主题/presentation.html`

1. 读取确认后的大纲
2. 读取 [slide-types.md](references/slide-types.md) 获取各组件的详细 HTML 结构
3. 读取选定的主题模板文件（如 `themes/ultimate-dark.html`）
4. 以主题模板为蓝本：保留 `<head>` 中的 `<style>` 和 `<script>`，保留 `#progressBar`、`.global-header`、`.global-footer`
5. 清空 `<div id="slides-wrapper">` 内部，替换为按大纲生成的 `<section class="slide">` 序列
6. 输出单文件零依赖 HTML

**⚠ 生成完成后，必须立即执行 `open YYYYMMDD_主题/presentation.html` 在浏览器中打开预览。不要跳过这一步。**

### 封面规则

每次生成 PPT 时固定第一页为封面（`type-cover`），包含：

- `name`：核心主题
- `series`：系列/大主题
- `tagline`：关键字标签
- `icon`：图标（如有）

### 渐变文字可用 class

`gradient-text-accent`（青紫渐变）、`gradient-text-cyan`（品红青渐变）、`gradient-text-warm`（黄红暖色渐变）

### 抽卡模式（多版本生成）

当用户在阶段 4 要求「生成多个版本」（如"抽5次卡"、"多来几版"）时，通过**不同大纲风格 × 不同视觉主题**的排列组合来制造差异：

1. 以当前大纲为基底，用不同的风格文件重写标题和内容表达（如乔布斯风版 + 简洁风版 + 叙事风版）
2. 为每个版本匹配不同的视觉主题（如版本1用 ultimate-dark，版本2用 notebook-tabs）
3. 输出多个 HTML（`presentation_1.html`、`presentation_2.html` 等）
4. 生成简单的 `index.html` 导航页，方便对比浏览
5. 可使用 subagent 并行处理每个版本

这样即使内容相同，**标题风格 + 视觉风格**的组合也能产生显著差异（而非 90% 相似的重复输出）。

**⚠ 阶段结束时：更新 `_progress.md`，记录生成的文件路径。下一步 → 阶段 5。**

## 阶段 5: 定制化视觉优化

基于已生成的标准组件 PPT，进行深度视觉与版式升级。这是拔高 PPT 设计感的关键环节。

> 🚨 **核心优化原则：打破常规，从零创新！**
> 在进入本阶段优化时，大模型必须**完全忘掉阶段 2 中的 17 个标准组件**！绝不能受原有组件结构、DOM 层级或原有 CSS 类的限制。必须**完全从 0 开始**构思和编写代码，大胆采用颠覆常规的非对称排版、沉浸式全屏图文、极致的空间留白、复杂的交互或高级动画图解等，打造出独一无二、高度定制化的视觉呈现。

### 优化执行流程：

1. **询问确认**：首先向用户确认“有没有已经确认想要优化的特定页面？”
2. **主动建议**：AI 结合当前 PPT 内容自行判断，提出针对性的优化建议。如果优化建议较多或跨多页，**必须先落地到文档**（如 `优化建议.md`）中与用户进行讨论确认。
3. **定制化重构**：获得用户确认后，严格遵循上述「核心优化原则」开始彻底修改和重构代码。

**⚠ 阶段结束时：更新 `_progress.md`，记录优化进度。下一步 → 阶段 6（如需口播稿）或阶段 7。**

## 阶段 6: 口播稿生成（可选）

仅在阶段 1 用户选择「需要口播稿」时执行。

**产出物**：`YYYYMMDD_主题/口播稿.md`

1. 基于**已生成的 PPT 页面结构**来写口播稿（而非先于 PPT）
2. 保持自然口语化，用「你/我/大家」拉近距离
3. 教程式开场：讲什么 + 为什么 + 流程预告
4. 实操演示位置标注 `*(动作：此处切到屏幕录制)*`
5. 每页附预估停留时间（按 220 字/分钟）

**⚠ 阶段结束时：更新 `_progress.md`，记录口播稿路径和预估总时长。下一步 → 阶段 7。**

---

## 阶段 7: 联合审核与优化

自动执行审核（详细规则见 [checker-rules.md](references/checker-rules.md)）：

1. **内容匹配度**：PPT 要点 ↔ 口播稿覆盖度
2. **节奏健康度**：
   - 单页停留 > 2分钟 → 🔴 过载，建议拆页
   - 连续多页 < 15秒 → 🟡 空窗，建议合并
3. **整体时长预估**
4. 输出精简审核报告，只列需要行动的问题
5. 如有可自动修复的问题（如拆页），直接执行；需确认的问题列出建议

**⚠ 阶段结束时：更新 `_progress.md`，标记全流程完成。告知用户所有产出物的路径。**

```

├── 大纲.md             # 确认后的最终大纲
├── presentation.html   # 生成的 PPT
