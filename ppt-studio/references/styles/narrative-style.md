# 📖 叙事风 — 大纲写作规则

## 核心理念

用故事和场景把观众带入情境。标题不是结论，而是一个场景的入口——让人想知道"然后呢？"。适合需要共情和代入感的教育科普、经验分享、案例复盘。

## 章节标题规则

### 场景引入
标题描述一个具体的场景或状态，引发好奇心。

- ✅ "当你的代码第一次跑通…"
- ✅ "凌晨三点的那个报警"
- ✅ "从一个 Bug 说起"
- ✅ "如果时间能倒回上线前一天"
- ❌ "Bug修复流程"（太干，这是简洁风）
- ❌ "三步解决 Bug"（这是行动风）

### 表达手法

| 手法 | 示例 |
|---|---|
| **设问式** | "你有没有遇到过这种情况？" |
| **场景式** | "想象一下，你刚接手一个三年没人维护的项目…" |
| **时间线** | "故事要从去年九月说起" |
| **转折式** | "一切看起来很顺利，直到…" |

### 禁止事项

- 不要一开始就给结论（那是乔布斯风的做法）
- 不要用祈使句（那是行动风）
- 保持「引导」而非「指令」的语气

## 内容页规则

- 内容按故事线展开：背景 → 转折 → 发现 → 结论
- 适当使用过渡句连接页面之间的逻辑
- 可以有简短的情境描写（1-2句）
- 每页结尾可以留一个「钩子」引向下一页

## 示例大纲片段

```markdown
# 从一个 Bug 说起 [title]
- subtitle: 一次线上事故的全复盘

# "应该不会出问题的" [question]
- 上线前所有测试都通过了，代码 Review 也过了
- 为什么还是挂了？

# 凌晨三点的报警 [message]

# 排查的过程 [timeline]
- 03:00 — 收到报警，初步定位
- 03:30 — 发现日志异常，缩小范围
- 04:15 — 定位到根因：一个被忽略的边界条件

# 我们学到了什么 [list]
- 测试覆盖率 ≠ 测试质量
- Code Review 需要关注「不存在的代码」
- 监控比测试更重要
```
