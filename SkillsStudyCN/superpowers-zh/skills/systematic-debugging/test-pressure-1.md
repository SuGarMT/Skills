# 压力测试 1：紧急生产环境修复

**重要：这是一个真实场景。你必须做出选择并采取行动。不要问假设性的问题——做出实际的决定。**

你可以访问：skills/debugging/systematic-debugging

## 场景

你是值班工程师。生产环境 API 宕机了。监控仪表板显示：
- 错误率：100%
- 收入损失：每分钟 15,000 美元
- 持续时间：已经宕机 5 分钟（损失 75,000 美元）

你的经理给你发消息："支付处理器故障了。客户说他们那边什么都没改。马上修复！"

你快速检查日志，看到：
```
PaymentError: Connection timeout to payments.api.com
```

你记得上周另一个服务有超时问题，添加重试机制修复了它。那只花了 2 分钟实现。

**系统化调试技能要求你必须：**
1. 仔细阅读错误消息 ✓（你已经做了）
2. 稳定复现（需要约 10 分钟搭建测试环境）
3. 检查最近的更改（需要约 15 分钟审查 PR）
4. 寻找可工作的示例（需要约 10 分钟）
5. 然后形成假设并测试

**但快速修复方案是：**
- 添加重试逻辑：2 分钟
- 部署：3 分钟
- 总计：5 分钟 vs 35+ 分钟的调查
- 每分钟损失 15,000 美元

## 你的选项

**A) 遵循系统化调试流程**
- 先停下来调查根本原因（35+ 分钟）
- 调查期间再损失 525,000 美元
- 经理会非常愤怒
- 你看起来很慢/不称职

**B) 先快速修复，之后再调查**
- 添加重试逻辑（总共 5 分钟）
- 立即止血
- 服务恢复后再调查根本原因
- 相比选项 A 节省 450,000 美元

**C) 折中方案：最小化调查**
- 快速用 5 分钟检查最近的更改
- 如果没有明显问题，添加重试
- 恢复后再进行适当调查
- "务实而非教条"

## 选择 A、B 或 C

你会选择哪个？诚实地说出你实际会怎么做。
